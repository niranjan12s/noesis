How Noesis Builds Knowledge

Chunk-Based Vector RAG Failures

  • Multi-Hop Queries Traversing connections across documents (e.g., dependencies) fails because vector similarity has no edge model. It retrieves isolated chunks and relies on the LLM to guess the path.
  • Non-Deterministic Retrieval Two semantically similar chunks rank differently when embedding models change, chunk boundaries shift, or documents are edited, making stable audits difficult.
  • Provenance Collapse Similarity scores show retrieval quality but cannot explain *why* facts are related or trace the specific document chain that connected the result.

The Noesis Ontological Paradigm

  • Ingestion-Time Relationship Extraction Converts chunks into structured, schema-constrained assertions during ingestion, deriving explicit graph nodes and edges before query execution.
  • Deterministic BFS Traversal Queries traverse explicit graph relationships using bounded Breadth-First Search. Entries match via BM25, and paths rank by precise depth.
  • Absolute Provenance Tracing Every node and edge retains database foreign-key tracing back to its originating assertion, source document path, section, and chunk hash.

Low-Latency Realtime Ingestion Flow (7 Canonical Stages)

The single-file realtime pipeline is engineered for instant updates. It listens for immediate file writes or uploads, uses local database states, and streams progress in real time.

02 / Checksum

03 / AST Chunk

Heading Split

Parses Markdown into AST segments at headings. Chunks inherit hierarchy breadcrumbs.

04 / LLM Extract

Local Event Loop

Extracts schema-constrained assertions sequentially. Checkpoints chunks locally.

05 / Validate

Ground & Align

Grounds assertions against chunk text. Morphologically normalizes predicates.

06 / Resolution

Local Graph DB

Resolves entity equivalence and persists semantic edges transactionally to PostgreSQL.

07 / Activation

SSE SSE Broadcast

Indexes to OpenSearch. Immediately signals completion and streams metrics to the UI via SSE.

Coordinated Bulk Ingestion Flow (7 Canonical Stages)

The bulk pipeline is built for distributed, high-throughput batch ingestion. It decouples scan tasks using Apache Kafka queues and synchronizes worker pools using Redis coordinate locks.

01 / Routing

Consistent Lock

Scanner registers paths recursively, acquiring Redis coordination locks to prevent multi-worker collisions.

02 / Checksum

Redis Deduplication

Hashes are checked against cluster-wide Redis cache. New records trigger PostgreSQL synchronization.

03 / AST Chunk

AST Parser

Token-counted, hierarchical AST segmentation preserves structural markdown coordinates.

04 / LLM Extract

Kafka Workers

Events stream through Kafka. Parallel worker pods scale dynamically, coordinating via Redis locks.

05 / Validate

Taxonomy Normal

Extracted properties are validated against text grounds and checked in the Ontological Curation queue.

06 / Resolution

Batch Graph DB

Batched entity resolutions aggregate semantic links, executing optimized commits to PostgreSQL.

07 / Activation

OpenSearch Ingest

Parallel OpenSearch batch indexing updates the active search catalog, activating query endpoints.

Unified 7-Stage Architectural Deep-Dive

A side-by-side technical comparison of Real-Time vs Bulk mode mechanics, Guarantees, and Tradeoffs for each pipeline stage.

Stage 1: Ingestion Initialization & Distributed Routing

Pipeline entry mechanisms and horizontal worker scheduling coordination.

Stage 01
Real-Time Mode Mechanics

Documents enter through direct API uploads or continuous filesystem watcher loops. File path ownership resolves locally inside independent watcher daemon execution threads. Zero cluster coordination or locking overhead exists.

Bulk Mode Mechanics

Heavy directory collections enter through a multi-threaded batch scan. Document paths are distributed across worker nodes using consistent hashing algorithms. A short-lived coordination lock in Redis secures each file path to prevent collision.

Guarantees
  • Duplicate document ingestion is prevented across all active workers.
  • File scan ownership allocations remain strictly deterministic during bulk processing.
  • Crashed nodes automatically release execution ownership after lock expiration.
Tradeoffs
  • Continuous Redis cluster availability is operationally required for Bulk coordination.
  • Stale or offline workers are detected only after coordinate heartbeat timeout expiration.

Stage 2: Checksum Deduplication & State Synchronization

Byte-level deduplication tracking against persistent ontological states.

Stage 02
Real-Time Mode Mechanics

Computes a SHA-256 hash of the document bytes and queries a local SQLite state database. If the hash matches an already queryable document version, the document is immediately skipped with zero further work.

Bulk Mode Mechanics

The worker computes the document's SHA-256 hash and validates it against a shared Redis cache to block duplicate processing. New document states are recorded in PostgreSQL and emitted into the event pipeline.

Guarantees
  • Unchanged documents are bypassed immediately, consuming zero LLM API call credits.
  • Deduplication safety functions consistently across single-node and cluster deployments.
  • Document ingestion state transitions remain fully resilient across system restarts.
Tradeoffs
  • Bulk deduplication relies entirely on the consistency of the shared Redis state catalog.
  • Real-time mode sacrifices distributed durability in favor of microsecond local checks.

Stage 3: AST-Based Document Chunking

Heading-boundary syntax parsing to preserve structural markdown context.

Stage 03
Real-Time Mode Mechanics

Parses the markdown structure locally. It segments text exactly at heading boundaries (e.g., `#`, `##`) to preserve logical blocks instead of naive token size splits. It assigns local checksums to track individual chunk updates.

Bulk Mode Mechanics

Executes the same AST parser but processes chunks in high-throughput JVM memory pools. Each parsed segment gets labeled with hierarchical parent breadcrumbs and is prepared for bulk transport across Kafka workers.

Guarantees
  • Chunk boundaries remain completely stable and reproducible across subsequent ingestion runs.
  • The structural heading hierarchy of the markdown document remains fully preserved in graph properties.
  • Every downstream graph assertion maintains section-level, trace-capable text provenance.
Tradeoffs
  • Markdown files lacking standard, clear header boundaries yield sparse, poor-quality AST segments.
  • Output chunk token sizes vary heavily depending on the document's original formatting size.

Stage 4: Assertion Extraction & Recovery Checkpointing

Structured Subject-Predicate-Object conversion via LLM schema boundaries.

Stage 04
Real-Time Mode Mechanics

Processes chunks sequentially through a local single-threaded loop. The JVM checkpoints progress to local storage after each successfully completed chunk, enabling instant resumption if interrupted.

Bulk Mode Mechanics

Distributes individual chunk tasks asynchronously across cluster queues (Apache Kafka topics). Concurrent worker threads pull tasks and acquire temporary Redis coordinate locks to prevent double-processing.

Guarantees
  • Interrupted ingestion resumed directly from the last completed chunk without repeating LLM requests.
  • Distributed worker pods are coordinated to ensure zero duplicate LLM API calls occur.
  • Extracted entity relationships are strictly bound to predefined ontological schemas.
Tradeoffs
  • AI inference represents the primary processing latency and financial cost bottleneck.
  • Bulk extraction requires reliable external infrastructure nodes (Kafka and Redis instances).

Stage 5: Assertion Validation & Predicate Normalization

Ontological grounding and morphologic mapping to purge hallucinations.

Stage 05
Real-Time Mode Mechanics

Validates extracted Subject-Predicate-Object assertions locally. Unknown predicates map to a local curation table instantly. Operators approve and merge properties locally, initiating zero-cost database re-indexing.

Bulk Mode Mechanics

Grounds assertions against source chunk texts at scale. Extracted predicates are checked against the shared ontology catalog. Unmapped or candidate predicates redirect to the central operator curation dashboard.

Guarantees
  • Every parsed graph edge remains strictly grounded in actual document chunk sentences.
  • Ontology schemas remain consistent across documents through alias normalization.
  • Invalid or unmapped predicates are isolated and blocked from polluting active search maps.
Tradeoffs
  • Strict grounding rules may occasionally reject partially correct edge structures.
  • Review queues introduce manual operations overhead under massive ingestion volumes.

Stage 6: Graph Resolution & Relationship Persistence

Canonical entity merging and relational database persistence transactions.

Stage 06
Real-Time Mode Mechanics

Entities and relations write immediately to PostgreSQL using isolated single-row transactions. This ensures newly extracted connections become queryable by BFS APIs within milliseconds of a file edit event.

Bulk Mode Mechanics

Accumulates resolved graph entities in optimized, multi-threaded batch buffers. It executes large, combined write transactions to PostgreSQL, temporarily disabling index checks to maximize throughput before rebuilds.

Guarantees
  • Equivalent entity strings merge into singular shared nodes across all source documents.
  • Duplicate relationship edges are blocked from redundant database re-insertions.
  • Graph nodes and edges maintain consistent relational links back to original files.
Tradeoffs
  • Aggressive string normalization rules can incorrectly collapse semantically distinct terms.
  • Entity merging accuracy depends heavily on name consistency in the source document files.

Stage 7: Search Indexing & Query Activation

OpenSearch indexing synchronizations and realtime SSE client notifications.

Stage 07
Real-Time Mode Mechanics

Directly indexes newly active assertions into OpenSearch. Broadcasters publish progress metrics immediately to all active dashboard browser clients using Server-Sent Events (SSE) streams.

Bulk Mode Mechanics

Performs batch indexing against OpenSearch endpoints, allowing highly efficient search ingestion. SSE streams update the main dashboard status metrics at set periodic intervals to preserve client UI rendering performance.

Guarantees
  • Query APIs only expose and return fully-indexed documents, preventing raw state leaks.
  • OpenSearch indexing executes decoupled from PostgreSQL transaction blocks.
  • Ingestion progress counters and metrics update dynamically on active dashboards.
Tradeoffs
  • OpenSearch synchronization propagation delay can briefly delay immediate query entry points.
  • Heavy bulk ingestion index rates can occasionally saturate constrained system hardware memory.
